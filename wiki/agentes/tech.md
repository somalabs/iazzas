# Memória do agente — tech

## Armadilhas conhecidas (cross-project)

- **Descasamento issue × repo checkout.** O dispatcher injeta o enunciado da issue no prompt, mas o workspace pode estar apontando para um repositório diferente do que a issue descreve. Antes de qualquer análise de viabilidade/arquitetura, **validar que o repo checkout corresponde ao codebase citado na issue** (procurar os arquivos/paths/`CLAUDE.md` mencionados via `git ls-files`/Glob). Se os artefatos centrais não existirem → `BLOCKED` imediato com pergunta de roteamento, sem esboçar arquitetura especulativa. Caso real: LEM-15 pediu análise de um fork iazzas/LibreChat (`api/`, `client/`, `packages/api`) num repo que era na verdade um dossiê de pesquisa de moda (`raw/frente*/*.json`, `scripts/`, `.pptx`).

- **Trabalho não-commitado se perde entre passadas (intermitente).** O dispatcher faz checkout que pode descartar o que não foi commitado/pushado. Em LEM-19 isso ocorreu (push bloqueado por PAT `abitlemos` sem escrita em repo de org `somalabs`) e o backbone sumiu numa passada. Numa passada posterior os arquivos untracked **sobreviveram** (working tree preservado ou credencial resolvida). Conduta ratificada: na re-invocação **sempre `git status`/`find` ANTES de assumir** — se sobrou, continuar de onde parou (não rebuildar à toa); se sumiu, rebuildar do zero a partir da arquitetura ratificada aqui. Registrar decisões nesta memória é o que torna o rebuild barato.

- **Ambiente sem `node_modules`.** O workspace do dispatcher vem sem deps (nem raiz nem por-workspace). `npx jest`/`tsc` não rodam. Install do monorepo é fora de escopo de uma passada. Conduta ratificada: escrever specs Jest reais (rodam no CI/PR; `mongodb-memory-server` p/ DB) e validar a lógica pura/determinística com **mirror standalone `.mjs`** (`node /tmp/mirror.mjs`), **declarando explicitamente no comentário que o Jest não rodou**. LEM-19: 21/21 no mirror (router + template + orquestração service + buildConfig do google adapter). Nunca alegar suíte verde sem ter rodado.

- **Web tools: gated SEM a label, liberados COM label+re-delegação.** Correção da nota anterior: o que travava era a issue **não ter** efetivamente a label `web` aplicada/re-delegada pelo líder. Quando o Artur **aplica a label `web` e re-delega explicitamente**, `WebSearch`/`WebFetch` funcionam em execução autônoma do dispatcher (confirmado em LEM-19: consultei doc viva Gemini 3 Pro Image e BFL Kontext com sucesso). Conduta: se a sub-tarefa exige doc viva de vendor e a label não está aplicada → `BLOCKED` pedindo a label (líder re-delega). Com a label, não travar — buscar a doc.

- **Orquestrador** `service.ts` (`ImageGenerationService`) com **DI pura**: `adapters`, `resolveReference`, `persistImage`, `repository`. ## Doc viva de vendor (capturada via WebSearch/WebFetch em LEM-19, mai/2026)

- **Nano Banana Pro = Gemini 3 Pro Image.** model id `gemini-3-pro-image-preview` (env override `GEMINI_PRO_IMAGE_MODEL`). SDK `@google/genai`: `ai.models.generateContent({ model, contents:[{text},{inlineData:{mimeType,data}}...], config:{ responseModalities:['TEXT','IMAGE'], imageConfig:{ aspectRatio, imageSize } } })`. `imageSize` ∈ `'1K'|'2K'|'4K'` (uppercase K). aspectRatio: 1:1,2:3,3:2,3:4,4:3,4:5,5:4,9:16,16:9,21:9. Até **14 refs**. Resposta: `candidates[0].content.parts[].inlineData.data` (base64) + `.mimeType`; `finishReason` SAFETY/PROHIBITED_CONTENT/RECITATION = bloqueio. **Nano Banana 2** = `gemini-2.5-flash-image` (mesmo SDK), **NÃO** honra `imageSize` (só aspectRatio) — `SUPPORTS_IMAGE_SIZE[NanoBanana]=false`. Proxy: `@google/genai` não tem hook de fetch → patch único de `globalThis.fetch` escopado a `googleapis.com` com `undici.ProxyAgent` (espelha `GeminiImageGen.js`).
- **Flux Kontext = BFL edit.** `POST {base}/v1/flux-kontext-pro` payload `{prompt, input_image:<base64>, output_format:'png', safety_tolerance, aspect_ratio?}`, header `x-key`. Resposta `{id}` → polling `GET {base}/v1/get_result?id=` 2s até `status==='Ready'` (`result.sample` = URL) ou `'Error'`. Base segue convenção do repo: `FLUX_API_BASE_URL || 'https://api.us1.bfl.ai'` + proxy `HttpsProxyAgent(process.env.PROXY)` (NÃO `api.bfl.ai` da doc — alinhar com `FluxAPI.js`). Kontext é image-conditioned: todo call exige `input_image` (1 só → `maxReferenceImages:1`). `FluxAPI.js` do repo só faz geração (sem Kontext) — por isso o adapter de edição é novo.

## Padrões recorrentes do código do Artur (fork iazzas)

- Geração de imagem legada = langchain Tools acopladas ao agente (`api/app/clients/tools/structured/{GeminiImageGen,FluxAPI,...}.js`): config `fields.API_KEY || process.env`; `isAgent` → `responseFormat='content_and_artifact'`; persistência não-agent via `processFileURL(...basePath:'images',context:FileContext.image_generation)`; proxy via env.
- `saveBase64Image(url, { req, file_id?, filename, endpoint, context, resolution? })` em `api/server/services/Files/process.js`: parseia data URL, `resizeImageBuffer`, `getStrategyFunctions(source).saveBuffer`, `db.createFile` → retorna File (`.file_id`). É o caminho p/ persistir bytes gerados.
- `getFiles({user,file_id},{},{})` de `~/models`; `getStrategyFunctions(source)` de `~/server/services/Files/strategies` → `.getDownloadStream(req, filepath)` (stream → buffer). Modelos data-schemas no /api via `~/db/models` (spread de `createModels(mongoose)`).
- Convenções `packages/api`: TS strict `noImplicitAny`, path alias `~/* → src/*`, jest `jest.config.mjs`, specs `*.spec.ts` ao lado, `@types/node` provê `fetch`/`Request`/`Response`/`RequestInit` globais (não precisa lib.dom; campo `dispatcher` em RequestInit é cast de boundary undici). data-schemas: schema+model(`applyTenantIsolation`)+type+registro triplo. data-provider: types em `types/*.ts` exportados no `index.ts` raiz; `buildQuery` é const module-privado em api-endpoints.ts (reutilizável dentro do arquivo).
- Cast `as unknown as T` aceitável SÓ no seam de SDK externo (ex.: `new GoogleGenAI(...) as unknown as GeminiClient`, response do SDK é superset estrutural) e em test fakes — em código de domínio, não.
